CTP : Network Application Readiness(NAR) - How To Guideline

Created by Lakshminarayanan Mukundagiri (TMNA), last modified by Unknown User (433661) on May 21, 2018

Introduction: 

This purpose of this document is to identify the NAR  process, environments, timelines, artifacts, result reporting and key contacts that has to be used for registration applications. The process will be documented and can be used when application is ready for NAR.

What is NAR:

The Network Application Readiness (NAR) analysis is used to assess how the Wireless Area Network (WAN) could affect user response times for an application and to find out how an application might impact the WAN and the performance of other applications sharing the network.  The results of the NAR test can also serve as a benchmark for later behavior analysis or troubleshooting once the application has moved to production. 

Why do we need that 

Typically a NAR involves identifying key transactions within a new application, collecting data related to those transactions, and analyzing the collected data.  At the end of the process, a NAR Memo is issued, with 'grades' of 'Pass'/'Fail'/'Attention Required' in two categories; WAN and User Experience.

  • The WAN grade is based on how the application affects the Toyota Network infrastructure.
  • The User Experience grade is based on how the application will behave across the network from the User’s point of view, with user response time predictions based on the locations where the application will be used.

In some cases, if a new application uses well-known processes, a data collection and analysis can be avoided, in which case a NAR Waiver memo will be issued to allow the application on the network.

 

What does NAR Testing Encompass

Those applications which are eligible for data collection and analysis can be divided into three broad categories:

(1)     Baseline only NAR - user experience is tested from the Data Center, with data stored for the tested user actions; a NAR memo is issued listing those user response times as tested.

  •  This effort usually will take 10-20 billable hours depending on the number of transactions under review.
  •  The data for the transactions are stored for possible later analysis, should that be found necessary.

(2)     Limited Analysis NAR - user experience is tested from the Data Center, with a data collection analyzed for a limited number of transaction steps; a NAR memo is created grading WAN Effect and User Experience for the steps, including predicted response times at different remote site types, such as Regions, Area Offices, PDCs, or the Internet.

  • This effort usually will take 25-40 billable hours depending on the number of transactions under review.

(3)     Full Analysis NAR - user experience is tested from the Data Center, with a data collection analyzed for all transaction steps; a NAR memo is created grading WAN Effect and User Experience for those steps including predicted response times for users at different remote site types.

  • This effort usually will take 40-52 billable hours depending on the number of transactions under review.

The decision as to which category of NAR analysis will be used for an application will be determined by the WAN team at the time the NAR is officially initiated. It is also unto the scope of the test scripts that we request NAR team to run for the transactions of our interest.

 NAR Process

The following are the typical steps for the NAR process:

(1)     A meeting should be scheduled by the project with WAN Engineering and WAN-Ops to discuss the scope of the NAR analysis required for the application/project (see #3, below).  Relevant documents, such as a Solution Architecture Document, Supplementary Specification, IDD, etc. should be made available for review before the meeting.

(2)     Complete the Discovery Questionnaire based on the template shared by NAR Team with the scope of the transactions to be executed in the syntax / format laid out

(3)     The project liaison and representatives from the project will meet with Network Design & Engineering and WAN-Ops engineers to discuss the application, determine the type of NAR required, which, if any, application transactions are to be included in a NAR data collection for analysis, and an estimate of the number of hours required for WAN-Ops.  The estimate will be based on the categorization of the NAR effort as described above, the number of application transactions to be analyzed and the number of individual steps within each transaction.

(4)     The project must identify and secure the participation of a ‘power user’ – a user thoroughly familiar with the application that can work with the network engineer.

(5)     The project will generate NAR scripts for the identified key transactions.  NAR scripts are step-by-step instructions for each significant task that form an application transaction.  With this memo, we will have provided a template spreadsheet provided.

As with the Questionnaire, copy this script and modify the title as appropriate for your project.  This spreadsheet will be used during the NAR data collection process – be sure to make it as specific and detailed as possible.  Please read the instructions in the script closely.  If appropriate, we will also ask that SLAs for each of the steps within the tested transactions be provided.

(6)     The sponsor and the power user will meet with the assigned network engineer to review the application and approve the transaction scripts.

(7)     It is important to stress that the application transactions under review must be working at the time of NAR data collection, and that the QA environment against which the scripts are to be tested is in a stable state.  To ensure that the testing can proceed without error, we require that the scripts are validated by the application team prior to the NAR collection and that the date of that validation is entered in the scripts to be used during the collection.

(8)     The power user and network engineer will meet in the Data Center lab (or other location as mutually agreed) to conduct the NAR data collection.   After the collection is complete, a copy of the completed test script(s), containing the recorded perceived user response times for the transaction steps as tested will be provided to the project on the day of the collection.  Please review this document immediately.

Note that the sponsor will be contacted prior to the generation of the final NAR document if problems are found during the analysis, to determine if modifications to the application may need to be made.  If necessary, a second NAR collection might need to be scheduled after such modifications.  The time required to conduct the document will vary depending upon the amount of data captured and the depth of analysis necessary.

(9)     A NAR document will be emailed to the sponsor when the NAR analysis is complete.  The document will indicate whether the application passed or failed the NAR criteria.  If necessary, a meeting may be set up to discuss the results of the NAR.

(10)     The management of the Business Unit that 'owns' the application will be required, after reviewing the results of the NAR, to issue an email indicating they understand and acknowledge the content of the report.  The reviewing approver must be direct BU management, Band 4 or Level 18 or above.                                                                                                                                            

NAR Readiness

  • Stable Code - BB Code Completed and No more version updates to the branch assigned for NAR testing
  • No Major functionality change - During the NAR test
  • No Significant architecture change - During the NAR test
  • Transactions should be tested before NAR

Environments

  • Stage (ONLY) - It is a billable project, need to be equipped to avoid multiple rounds of NAR testing.

TimeFrames

  • Request to be made 3 weeks in advance
  • Testing time depending on application/ testing complexity (Data collection - 1 week and Report - 2 Weeks)

Requirements

  • Define Transactions(Critical and Non critical)
  • Transactions SLAs
  • Application walk through with NAR tester
