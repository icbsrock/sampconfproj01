CTP : Value Stream Mapping - Current

Created by Unknown User (430384), last modified on May 19, 2018

Objective

Value Stream Mapping is a crucial step in assessing an organization’s DevOps capability. We create a value stream map of the software development lifecycle early in any DevOps engagement because it helps us to:

  • Provide context for our architects and technical experts who are, in parallel, examining tools and technology.
  • Establish a performance baseline, against which the improvements delivered by adopting DevOps will be measured. 
  • Begin the process of winning hearts and minds, and engender cultural change. 
  • By working side-by-side in a workshop, Dev and Ops resources, as well as those involved in planning, test scheduling, and other activities, start to gain an understanding of how the work they do is linked to steps upstream and downstream. 
  • Building a Value Stream Map lays the groundwork for closer coupling of those activities in the future.

How do we go about it

We ask the same questions across all these types of processes – and now we ask those questions of the Software Development Lifecycle. Those questions are:

  • What activities are being performed?
  • Who delivers those activities? How much of their time is dedicated to those activities?
  • What is the yield, i.e. ‘right first time’? 
  • What % of rework is generated? 
    • What causes rework?
  • What tools are required to complete the step? 
    • What other inputs are required?
  • How many units (e.g. work packages) are ‘work in progress’ within this step?
  • How many units are waiting in the ‘inbox’? 
    • How will they be prioritized or scheduled? 
    • How long will they, typically, wait for?
  • How long does the step take? 
    • How much of that time is the effort, or touch time, versus duration, or in-process lead-time?

Why do we need to assess the Current State

We work to understand the current state and all define the steps currently performed in order to hear out loud what is missing or what is causing the utmost pain.  This is done to assist the team to increase the effectiveness and productivity of the overall team, i.e. to optimize the flow of the entire system. A high-level value stream mapping workshop was completed on May 17, 2018.  The value stream map and pain points that were uncovered as a result of the process.

Pain Points Identified

  • No versioning tags are used (in what?)

    • Software releases need embedded data so that its history can be tracked through the development process.

  • No tagging used to control the release

  • No gating for Sonarqube code issues

  • No gating for auto-deployment to staging and production

  • No notification for build deployment

  • No daily commits (internal kaizen)

  • It is not clear how Spinnaker gets the images from Artifactory

  • Currently, there is no dynamic or static scanning

  • No performance testing

  • We are getting incorrect build statuses 

    • At least 6 build statuses were misreported April

  • No automated regression tests

  • No automated security scanning (for example, Veracode static code analysis)
  • No visibility to how network provisioning happens in Spinnaker
  • If there is a failure in the pipeline build, CT opens a ticket in JIRA with the Nozomi team regarding the incident
  • Need to understand how dependency management issues are resolved in the Nozomi infrastructure
  • No performance testing is taking place
  • Recovery of a failed production deployment due to a simple coding issue requires 45 minutes on average to resolve

Additional Questions

  • Does the TC Artifactory instance have the JFrog XRay scanner analyzing the code packages?
  • How do we manage whitelists within the TC Nozomi environment?
